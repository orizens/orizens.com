# When React Hooks Stop Scaling: Moving Complex State to Zustand

For years, React Hooks have been my default choice for managing component state.

Most of the time, they're exactly what you need.

A `useState` here, a `useReducer` there, maybe a custom hook when things become reusable.

Simple.

But recently I found myself working on a feature that continuously processed speech recognition events, managed transcripts, tracked user speaking status, handled errors, and coordinated updates across multiple UI components.

At first, a custom hook seemed like the right abstraction.

Eventually, it became the problem.

## The Hook That Knew Too Much

The hook started innocently:

```tsx
const {
  transcript,
  error,
  isUserSpeaking
} = useSpeechRecognition();
```

Over time, more responsibilities were added:

* Managing speech recognition lifecycle
* Handling browser-specific behavior
* Updating transcripts
* Tracking speaking state
* Error management
* Synchronizing updates with different parts of the application

The API remained simple.

The implementation did not.

The hook became a mini application hidden behind a single function call.

## The Hidden Cost of Large Custom Hooks

One thing I've noticed is that React hooks make complexity easy to hide.

That's both their strength and their weakness.

As long as the hook works, nobody questions what's inside.

Then one day you need:

* Another component to access the same state
* State updates from outside React components
* Better debugging
* Shared state across multiple screens

Suddenly the hook is doing far more than React originally intended.

You start passing callbacks around.

You introduce refs.

You add more effects.

You add comments explaining why a specific dependency array should never be touched.

The complexity compounds.

## Why Zustand Was a Better Fit

The turning point came when I stopped thinking about the problem as a hook problem.

It was actually a state management problem.

The speech recognition logic wasn't tied to a component.

It represented application state.

That's where Zustand fit naturally.

Instead of hiding everything inside a hook:

```tsx
const useRecognitionStore = create(() => ({
  transcript: "",
  error: null,
  isUserSpeaking: false
}));
```

Now any component can subscribe only to the data it needs:

```tsx
const transcript = useRecognitionStore(
  state => state.transcript
);
```

The recognition service updates the store directly.

Components simply render state.

The responsibilities become clearer.

## Fewer Re-renders, Cleaner Components

One benefit that immediately became visible was component simplicity.

Before:

```tsx
const {
  transcript,
  error,
  isUserSpeaking
} = useSpeechRecognition();
```

After:

```tsx
const transcript = useRecognitionStore(
  state => state.transcript
);
```

Each component consumes exactly what it needs.

No large objects.

No unnecessary subscriptions.

No leaking implementation details.

## The Real Advantage Isn't Performance

Most discussions around Zustand focus on performance.

That's nice.

But for me, the bigger benefit is architecture.

The state becomes explicit.

The data flow becomes easier to understand.

The business logic becomes easier to test.

And perhaps most importantly, you stop building giant hooks that gradually evolve into hidden state containers.

## When I Still Prefer Hooks

This doesn't mean every hook should become a Zustand store.

If state is local to a component:

```tsx
const [isOpen, setIsOpen] = useState(false);
```

Keep it there.

If a custom hook encapsulates reusable UI behavior, that's also fine.

The question I ask myself now is:

> Would this state still make sense if React disappeared?

If the answer is yes, it's probably application state.

And application state deserves a dedicated home.

## Final Thoughts

React Hooks remain one of the best additions to React.

But not every state problem should be solved with another custom hook.

When a hook starts coordinating multiple systems, managing shared data, and acting as the source of truth for large parts of your application, it's worth stepping back and reevaluating the design.

In my experience, that's often the moment where Zustand starts making more sense than another layer of abstraction around `useState`.

The goal isn't to use fewer hooks.

The goal is to keep complexity visible, intentional, and easy to reason about.
